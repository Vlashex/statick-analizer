# Архитектура MVP статического анализатора сложности C++

## 1. Цель системы

Система оценивает асимптотическую временную сложность функций в `.cpp` файле без выполнения кода. Результат выводится в консоль в формате:

- имя функции
- оценка сложности

## 2. Scope MVP

В MVP реализуется только базовый сценарий:

1. запуск из CLI
2. чтение одного `.cpp` файла
3. токенизация
4. построение упрощенного дерева
5. расчет сложности по простым правилам
6. печать результата в консоль

В MVP не входят:

- API, веб, GUI, БД
- внешние сервисы и интеграции
- сложная конфигурация
- анализ всего стандарта C++

## 3. Архитектурные принципы

1. Minimum viable architecture: только необходимые модули для сценария `input -> analyze -> output`.
2. Domain first: правила сложности и доменная модель изолированы от деталей CLI.
3. Простые зависимости: `main` оркестрирует вызовы модулей напрямую, без DI и интерфейсных слоев.
4. Предсказуемость: одинаковый вход всегда дает одинаковый результат.
5. Без архитектурной избыточности: только модули, нужные для текущего CLI-сценария.
6. Строгие контракты: каждый модуль имеет фиксированный вход и выход и не использует внутренние структуры соседних модулей.

## 4. Общий pipeline анализатора

```text
CLI args -> FileReader -> Lexer -> SimpleParser -> ComplexityAnalyzer -> ConsolePrinter
```

Шаги выполнения:

1. CLI получает путь к `.cpp`.
2. Чтение исходного текста файла.
3. Лексер разбивает текст на токены.
4. Парсер строит упрощенное внутреннее дерево функций и блоков.
5. Анализатор рассчитывает сложность каждой функции.
6. Результат печатается в stdout.

## 5. Архитектура (DDD + Hexagonal как будущая модель)

### 5.1. DDD в рамках MVP

DDD в MVP используется только как логическое разделение:

- `model` хранит сущности анализа
- `analyzer` хранит правила расчета сложности
- `main` и `cli` только запускают pipeline

### 5.2. Совместимость с будущей hexagonal архитектурой

В MVP не вводятся порты, адаптеры и отдельные слои. Достаточно логической границы:

- ввод: CLI + чтение файла
- анализ: lexer + parser + analyzer + model
- вывод: печать в консоль

## 6. Минимальная структура проекта

```text
src/
  main.cpp
  cli/
  lexer/
  parser/
  analyzer/
  model/
```

Назначение каталогов:

- `cli`: разбор аргументов и формат консольного вывода
- `lexer`: токенизация исходного текста
- `parser`: построение упрощенного дерева
- `analyzer`: правила расчета сложности
- `model`: доменные структуры данных

## 7. Модель домена анализатора

Ключевые сущности:

- `InputPath`
- `SourceText`
- `Token`
- `TokenList`
- `TokenType` (`identifier`, `keyword`, `number`, `operator`, `brace`, `parenthesis`, `semicolon`)
- `FunctionNode`
- `LoopNode`
- `BranchNode`
- `BlockNode`
- `StatementNode`
- `FunctionNodeList`
- `Complexity` (например, `O(1)`, `O(n)`, `O(n^k)`)
- `FunctionReport`
- `FunctionReportList`

Ключевые слова лексера:

- `for`
- `while`
- `do`
- `if`
- `else`
- `switch`
- `return`

## 8. Компоненты анализатора

1. `CliRunner`
- принимает аргументы командной строки
- запускает pipeline

2. `FileReader`
- читает `.cpp` файл в строку

3. `Lexer`
- преобразует исходный код в список токенов

4. `SimpleParser`
- строит упрощенное дерево из токенов
- поддерживает конструкции `for`, `while`, `if` и блочную структуру `{ }`

5. `ComplexityAnalyzer`
- вычисляет сложность функции

6. `ConsolePrinter`
- печатает отчет в stdout

### 8.1. Контракты модулей (обязательно)

| Модуль | Вход | Выход |
| --- | --- | --- |
| `CliRunner` | `argc/argv` | `InputPath` |
| `FileReader` | `InputPath` | `SourceText` |
| `Lexer` | `SourceText` | `TokenList` |
| `SimpleParser` | `TokenList` | `FunctionNodeList` |
| `ComplexityAnalyzer` | `FunctionNodeList` | `FunctionReportList` |
| `ConsolePrinter` | `FunctionReportList` | stdout |

Единый формат передачи данных между модулями:

- вход и выход модулей описываются только типами из `model`
- модуль получает данные целиком из предыдущего шага и возвращает полный результат своего шага
- модуль не обращается к состоянию другого модуля напрямую

### 8.2. Правила независимости модулей

1. Только `main.cpp` собирает pipeline и вызывает модули по порядку.
2. `lexer` не зависит от `parser`, `analyzer`, `cli`.
3. `parser` не зависит от `analyzer`, `cli`.
4. `analyzer` не зависит от `lexer`, `cli`, `FileReader`.
5. `cli` не зависит от внутренних структур `lexer/parser/analyzer`.
6. Межмодульный обмен выполняется только через контрактные типы `model`.

## 9. Алгоритм анализа сложности

Для каждой функции:

1. Определить максимальную глубину вложенности циклов.
2. Для ветвлений `if/else` вычислить сложность каждой ветки.
3. Выбрать худшую ветку как вклад ветвления.
4. Скомбинировать результаты по правилу Big-O (доминирует более растущая часть).

Правила MVP:

- нет циклов -> `O(1)`
- один цикл -> `O(n)`
- глубина циклов `k` -> `O(n^k)`
- `if/else` -> максимум из веток

Отчет по функции включает:

- `Complexity`

## 10. CLI интерфейс

Формат запуска:

```bash
analyzer <path-to-file.cpp>
```

Формат вывода:

```text
Function: foo
Complexity: O(n^2)
```

Ошибки CLI:

- неверные аргументы
- файл не найден или недоступен
- некорректный вход для упрощенного парсера

## 11. Ограничения MVP

- анализ только одного файла за запуск
- поддерживается ограниченное подмножество C++
- анализ основан на структуре кода, без выполнения программы
- вызовы функций и рекурсия не решаются как отдельная задача
- `switch` допускается как токен, но сложный анализ случаев не обязателен

## 12. План расширения

1. Расширить покрытие парсера по конструкциям C++.
2. Добавить более точные правила анализа сложности.
3. Поддержать анализ нескольких файлов за один запуск.
