# Функциональная спецификация MVP статического анализатора сложности C++

## 1. Назначение

Анализатор оценивает асимптотическую временную сложность функций C++ без выполнения программы.

Целевой сценарий MVP:

`CLI -> чтение .cpp файла -> анализ -> вывод в консоль`

## 2. Scope MVP

В рамках MVP система должна:

1. принимать путь к одному `.cpp` файлу из аргументов командной строки
2. читать содержимое файла
3. выполнять упрощенный лексический и структурный разбор
4. рассчитывать сложность функций по базовым правилам
5. печатать результат в stdout

В рамках MVP система не должна:

- поддерживать весь синтаксис C++
- выполнять код
- анализировать внешние библиотеки
- использовать API, GUI, БД, сервисы и плагины

## 3. Входные данные

### 3.1. Формат запуска

```bash
analyzer <path-to-file.cpp>
```

### 3.2. Формат входа

- один файл `.cpp`
- текст на C++ с поддерживаемым подмножеством конструкций

### 3.3. Поддерживаемые конструкции

- функции
- вложенные функции
- циклы `for`
- ветвления `if`, `else`, `switch`
- простые операторы и выражения внутри блоков

## 4. Выходные данные

Для каждой найденной функции выводится:

- имя функции
- оценка сложности (`O(1)`, `O(n)`, `O(n^k)`)

Пример формата:

```text
Function: foo
Complexity: O(n^2)
```

## 5. Этапы анализа

### 5.1. Tokenization

Код разбивается на токены.

Поддерживаемые типы токенов:

- `identifier`
- `keyword`
- `number`
- `operator`
- `brace` (`{` `}`)
- `parenthesis` (`(` `)`)
- `semicolon`

Поддерживаемые ключевые слова:

- `for`
- `do`
- `if`
- `else`
- `switch`
- `return`

### 5.2. Построение упрощенного дерева

Полный AST C++ не строится.

Парсер формирует внутреннее дерево для анализа со следующими типами узлов:

- `FunctionNode`
- `LoopNode`
- `BranchNode`
- `BlockNode`
- `StatementNode`

Парсер ориентируется на:

- парные скобки `{ }`
- объявления функций, включая вложенные функции
- конструкции `for (...)`
- конструкции `if (...)`

### 5.3. Анализ сложности

Для каждой функции применяется упрощенный алгоритм:

- нет циклов -> `O(1)`
- один цикл -> `O(n)`
- вложенные циклы глубины `k` -> `O(n^k)`
- `if/else` -> выбирается худшая по сложности ветка

Результат анализа:

- итоговая оценка сложности

## 6. Функциональные требования

1. Система должна завершать анализ одного файла за один запуск CLI.
2. Система должна находить функции в поддерживаемом подмножестве синтаксиса.
3. Система должна поддерживать вложенные функции и строить дерево функций по областям видимости.
4. Система должна определять глубину вложенности циклов в функции.
5. Система должна учитывать ветвления по правилу худшего случая.
6. Система должна печатать отчет по каждой функции в консоль.
7. При неполном разборе система должна завершаться контролируемо и выводить диагностическое сообщение.
8. Модули должны взаимодействовать только через фиксированные входы и выходы каждого этапа pipeline.
9. Контракт этапов должен быть строгим: `SourceText -> TokenList -> FunctionNodeList -> FunctionReportList`.
10. Только `main` имеет право связывать модули в цепочку выполнения.
11. Модули не должны использовать внутренние структуры соседних модулей напрямую.

## 7. Ограничения MVP

- анализ выполняется только в пределах одной функции
- рекурсивные функции и рекурсивные вызовы не поддерживаются
- межфункциональные вызовы вне текущего дерева вложенности не анализируются
- `switch` допускается как токен и ветвление, без продвинутого анализа `case`
- точные оценки вида `O(log n)` и `O(n log n)` в MVP не являются обязательными

## 8. Критерии готовности MVP

MVP считается готовым, если:

1. программа запускается из CLI с путем к `.cpp`
2. для тестовых примеров без циклов, с одним циклом и с вложенными циклами выдаются ожидаемые классы сложности
3. вывод читается напрямую в терминале без дополнительных инструментов
